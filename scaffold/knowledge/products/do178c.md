# DO-178C wolfCrypt Certification

> One-line summary: Same wolfSSL codebase, different build config -- what code paths are in scope, what build flags matter, and MISRA-C implications.

**When to read**: Working on code that touches DO-178C build paths, MISRA-C compliance, safety-critical RTOS ports, or avionics customer questions.

---

## Scope: wolfCrypt Only, DAL A

Certification covers **wolfCrypt** (crypto engine), not the TLS layer. DAL A -- highest level (catastrophic failure condition). Certified algorithms: AES, SHA-256/384/512, RSA, ECC (P-256, P-384, P-521), HMAC, RNG.

## Same Codebase, Build-Time Distinction

NOT a fork. Same wolfSSL source, safety-critical build configuration:

| Define | Meaning |
|--------|---------|
| `HAVE_DO178` | Enables DO-178 safety checks and code paths (set by `--enable-do178`) |
| `NO_DO178` | Disables DO-178 paths (default; explicitly set in FIPS `user_settings.h` files) |

The `--enable-do178` configure option **requires a DO-178 commercial license** to function.

## Mandatory Build Flags for Safety Config

| Flag | Purpose |
|------|---------|
| `WOLFSSL_NO_MALLOC` + `XMALLOC_USER` | All memory caller-provided; eliminates heap failure modes |
| `NO_FILESYSTEM` | Certs/keys from memory buffers, not files |
| `TFM_TIMING_RESISTANT` | Constant-time math (side-channel protection) |
| `ECC_TIMING_RESISTANT` | Constant-time ECC |
| `WC_RSA_BLINDING` | RSA blinding against timing attacks |

These overlap with embedded best practices but are **mandatory** for DO-178.

## MISRA-C Implications

- MISRA-C:2012 enforced via static analysis (cppcheck, Coverity, clang-tidy, dedicated MISRA checkers)
- `sp_int.c` is the primary MISRA surface -- received specific compliance updates
- Documented deviations with rationale are expected (zero deviations is not the goal)

## Safety-Critical RTOS Ports

| RTOS | Define | Notes |
|------|--------|-------|
| DDC-I Deos | `WOLFSSL_DEOS` | Most constrained: `WOLFSSL_NO_MALLOC`, `NO_FILESYSTEM`, `deos_time()` |
| Green Hills INTEGRITY | -- | wolfBoot target on Intel Tiger Lake |
| VxWorks / LynxOS / PikeOS | -- | Additional safety-critical targets |

FIPS 140 + DO-178C dual builds are possible: same source, standard FIPS defines `NO_DO178`, dual build enables both.

## Known Failure Modes

| Symptom | Root Cause | Key File/Function |
|---------|------------|-------------------|
| Build fails with `--enable-do178` | No DO-178 commercial license | Configure script check |
| MISRA violation in `sp_int.c` | Math library is the primary MISRA surface | `wolfcrypt/src/sp_int.c` |
| Heap-related failure in safety build | `WOLFSSL_NO_MALLOC` not set or bypassed | `user_settings.h` |
| DO-178 paths active in FIPS build | `NO_DO178` not defined in FIPS `user_settings.h` | XCODE-FIPSv5/v6 settings |
