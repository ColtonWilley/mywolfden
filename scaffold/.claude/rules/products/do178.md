# DO-178C wolfSSL Patterns

## What Is DO-178C

RTCA DO-178C ("Software Considerations in Airborne Systems and Equipment Certification") is the standard that governs how software is developed and verified for use in airborne systems. It is recognized by the FAA (United States), EASA (Europe), and TCCA (Canada) as the means of compliance for software certification in commercial and military aviation. DO-178C replaced DO-178B in 2012 and added supplements for model-based development, object-oriented technology, and formal methods.

Software is categorized by Design Assurance Level (DAL) based on the severity of failure conditions: DAL A (catastrophic), DAL B (hazardous/severe-major), DAL C (major), DAL D (minor), and DAL E (no effect). Higher DAL levels require more rigorous verification. DAL A requires Modified Condition/Decision Coverage (MC/DC) — every condition in every decision must be shown to independently affect the outcome.

## wolfSSL DO-178C Certification

wolfCrypt (wolfSSL's cryptographic engine) has completed DO-178C certification at DAL A, the highest assurance level. This means the cryptographic library has undergone the full certification lifecycle including requirements-based testing, structural coverage analysis (statement, decision, and MC/DC coverage), and multiple Stage of Involvement (SOI) audit cycles with a Designated Engineering Representative (DER).

Key facts:
- wolfCrypt is certified at **DAL A** (catastrophic failure condition level)
- wolfCrypt is actively deployed and "in the air" protecting avionic systems
- The certification applies to **wolfCrypt** (the crypto engine), not the TLS layer — avionics customers typically use wolfCrypt directly for cryptographic operations
- wolfBoot (secure boot) leverages wolfCrypt DO-178 for its cryptographic functionality and has been ported to Intel Tiger Lake (11th Gen Core i7) with Green Hills INTEGRITY as the initial target OS

## Same Codebase, Different Build Configuration

This is NOT a separate fork or rewrite. DO-178C wolfSSL is the same wolfSSL codebase built with a safety-critical configuration. The distinction is controlled at build time:

- **`--enable-do178`** in configure (autotools) sets `-DHAVE_DO178`
- Without it, `-DNO_DO178` is defined by default
- The configure help string explicitly states: "Will NOT work w/o DO178 license"
- FIPS user_settings.h files (XCODE-FIPSv5, XCODE-FIPSv6, WINCE) explicitly `#define NO_DO178` to disable DO-178 code paths in standard FIPS builds

The DO-178 build enables additional safety checks, assertion coverage, and code paths that satisfy the certification requirements. Customers receive the same source tree plus certification artifacts (see below).

## MISRA-C Compliance

wolfSSL's codebase undergoes MISRA-C analysis as part of the DO-178C process. The SP math library (`sp_int.c`) specifically received MISRA-C compliance updates (noted in the wolfSSL ChangeLog). MISRA-C:2012 is the coding standard used — it defines rules to avoid undefined behavior, implementation-defined behavior, and common C pitfalls that are unacceptable in safety-critical contexts.

MISRA-C compliance is checked via static analysis tooling (cppcheck, Coverity, clang-tidy, and dedicated MISRA checkers). The Renesas CS+ IDE project files in the wolfSSL tree show MISRA-2012 analysis configuration, indicating the codebase is regularly verified against these rules even outside the DO-178 build.

Customers should understand that MISRA compliance does not mean zero deviations. Documented deviations with rationale are standard practice and are part of the certification artifacts.

## Safety-Critical RTOS Support

wolfSSL includes explicit platform support for safety-critical real-time operating systems used in avionics:

- **DDC-I Deos** (`WOLFSSL_DEOS`): Full port with custom memory allocation (`XMALLOC_USER`, `WOLFSSL_NO_MALLOC`), no filesystem (`NO_FILESYSTEM`), timing-resistant math (`TFM_TIMING_RESISTANT`, `ECC_TIMING_RESISTANT`, `WC_RSA_BLINDING`), ECC curves (P-192 through P-521), and TLS extensions. Uses `deos_time()` for time services.
- **Green Hills INTEGRITY**: Target OS for wolfBoot on Intel Tiger Lake
- **Wind River VxWorks**: Widely used avionics RTOS, supported by wolfSSL
- **LynxOS** and **SYSGO PikeOS**: Additional safety-critical RTOS targets

The Deos configuration is notable for `WOLFSSL_NO_MALLOC` — all memory comes from caller-provided buffers, eliminating heap-related failure modes that would complicate certification.

## Build Configuration Details

When building for DO-178C, the safety-critical configuration typically includes:

- **No dynamic allocation**: `WOLFSSL_NO_MALLOC` with `XMALLOC_USER` — all memory provided by the caller, eliminating heap fragmentation and allocation failure as failure modes
- **No filesystem**: `NO_FILESYSTEM` — certificates and keys loaded from memory buffers, not files
- **Timing-resistant operations**: `TFM_TIMING_RESISTANT`, `ECC_TIMING_RESISTANT`, `WC_RSA_BLINDING` — constant-time math to prevent side-channel attacks
- **Deterministic error handling**: Every function returns explicit error codes; no silent failures or undefined behavior paths
- **Static analysis clean**: Code paths enabled by `HAVE_DO178` are verified against MISRA-C:2012 rules

These settings overlap significantly with embedded best practices but are mandatory (not optional) for the DO-178 configuration.

## FIPS 140 + DO-178C Dual Certification

Some avionics customers require both FIPS 140-2/140-3 validation (cryptographic module correctness) and DO-178C certification (software safety assurance). wolfSSL supports this dual-certification path because:

- wolfCrypt FIPS and wolfCrypt DO-178 share the same underlying source code
- The FIPS boundary and DO-178 safety boundary can coexist in a single build
- Standard FIPS builds define `NO_DO178` to exclude DO-178 paths, but a dual build can enable both

This is relevant because government and military aviation programs often mandate both standards. The dual path avoids having to maintain two separate crypto libraries.

## Certification Artifacts

DO-178C certification is not just about code — it requires a substantial documentation package. wolfSSL provides:

- **Plan for Software Aspects of Certification (PSAC)** — top-level document describing the certification approach
- **Software Development Plan (SDP)** — processes, standards, and tools used
- **Software Verification Plan (SVP)** — test strategy, coverage criteria
- **Software Requirements** — traceable requirements for each cryptographic function
- **Test Cases and Procedures** — requirements-based tests with full traceability
- **Structural Coverage Analysis** — statement, decision, and MC/DC coverage reports
- **Configuration Management records** — full change history and baseline identification
- **Software Accomplishment Summary (SAS)** — compliance summary for the certification authority

These artifacts are provided under the DO-178 commercial license. The source code itself is the same open-source wolfSSL codebase — the certification value is in the artifacts, processes, and verified configuration.

## Common Support Questions

**"Can I use the open-source wolfSSL for DO-178C?"**
The source code is publicly available, but the certification artifacts, verified configuration, and ongoing support require a commercial DO-178 license. Without the artifacts you cannot claim DO-178C compliance to a DER. Contact facts@wolfssl.com.

**"Does the DO-178 variant cover TLS or just crypto?"**
The DO-178C certification covers wolfCrypt (the cryptographic engine). Avionics customers typically use wolfCrypt directly for encryption, hashing, signatures, and key agreement rather than the full TLS stack.

**"Which algorithms are included?"**
The certified configuration includes AES, SHA-256/384/512, RSA, ECC (P-256, P-384, P-521), HMAC, and random number generation. The exact algorithm set depends on the certified baseline — contact wolfSSL for the current list.

**"What DAL level does this support?"**
DAL A — the highest level, required when software failure could cause or contribute to a catastrophic failure condition.

**"Can I modify the source code and keep certification?"**
Modifications to certified code require re-verification proportional to the change. Small configuration changes (enabling/disabling algorithms) have defined processes. Structural changes to the crypto implementation would require significant re-verification effort.

**"What about wolfBoot for secure boot?"**
wolfBoot uses wolfCrypt DO-178 for cryptographic operations and has been ported to Intel Tiger Lake with Green Hills INTEGRITY. It is designed to enter the SOI (Stage of Involvement) process for its own DO-178C certification.

**"How does DO-178C differ from DO-178B?"**
DO-178C (2012) superseded DO-178B (1992). The core objectives are the same, but DO-178C added technology supplements: DO-331 (model-based development), DO-332 (object-oriented technology), DO-333 (formal methods). wolfSSL's certification is under the current DO-178C standard.

**"What is the SOI process?"**
Stage of Involvement (SOI) audits are review points where a DER examines the software lifecycle data. SOI 1 reviews plans, SOI 2 reviews development, SOI 3 reviews verification, and SOI 4 is the final review. wolfCrypt has completed multiple SOI audit cycles.
