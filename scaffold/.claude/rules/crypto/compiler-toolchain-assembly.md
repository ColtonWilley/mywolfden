# Compiler & Toolchain Assembly Investigation Guide

This guide teaches investigation patterns for compiler-specific assembly issues. It tells you WHERE TO LOOK and WHAT TO VERIFY — not what the answers are. Follow these patterns against the actual code to reach conclusions.

## wolfSSL's Multi-Compiler Assembly Output

wolfSSL generates assembly in three output formats: `.S` (AT&T, GCC/Clang), `.asm` (MASM, MSVC), and `_c.c` (C inline assembly, IAR/Keil/others). The `WOLFSSL_ARMASM_INLINE` define gates the `_c.c` path. The `_c.c` files have compiler-specific preambles — read the first 40-50 lines to see what compatibility mechanisms are active.

## Scripts Repo — Source of Truth for Generated Assembly

All assembly is generated by Ruby scripts in the `wolfssl/scripts` repo (searchable as "scripts" product). Architecture modules in `asm/thumb2/`, `asm/arm64/`, etc. provide platform-specific output logic. Generated files are checked into the wolfSSL source tree.

**Investigation pattern:** When assembly behaves differently across compilers, read the architecture's output module (e.g., `asm/thumb2/thumb2_c.rb`) to understand the INTENT behind generated patterns. The generator shows why register variables are assigned a certain way, why constants are ordered, or why there are separate compiler code paths.

## Compiler Dialect Investigation Patterns

IAR, Keil, GCC, and Clang each handle inline assembly syntax differently. wolfSSL's generated code handles this via preprocessor blocks in the `_c.c` file headers.

**Key compatibility mechanisms to check:**
- **`__asm__` keyword mapping** — GCC uses `__asm__`, others need remapping (e.g., `#define __asm__ asm` for IAR).
- **`register ... __asm__("rN")` syntax** — GCC extension for register binding. Look for `WOLFSSL_NO_VAR_ASSIGN_REG` when not supported.
- **Label syntax** — Naming conventions differ across compilers and versions. Check `#if defined(__IAR_SYSTEMS_ICC__)` blocks around labels.

**Investigation pattern:** Read the first 50 lines of the `_c.c` file for the compiler detection block. If the compiler IS handled, trace the generator for assumption bugs. If NOT handled, check the generator's `header()` method for coverage gaps.

## GCC Extended Inline Assembly Correctness Model

When investigating inline assembly bugs, you need to understand how the `asm volatile` block communicates register usage to the compiler. This is the foundation for diagnosing clobber list, operand constraint, and optimization-dependent bugs.

**Operand constraints — classify every operand:**
- `"r"` (input) — Compiler picks a register and loads the value. The asm body reads it via `%[name]`. The compiler assumes this register is NOT modified by the asm.
- `"+r"` (read-write) — Same register for input AND output. Compiler loads the value before the asm and reads it back after. The compiler knows this register IS modified.
- `"=r"` (output-only) — Compiler assigns a register for the asm to write. The asm body need not read it first.
- `"=&r"` (early-clobber output) — Output register is guaranteed DIFFERENT from all input registers. Use when the asm writes the output before reading all inputs. Without `&`, the compiler may assign the output to the same register as an input.

**The clobber contract:**
The clobber list tells the compiler: "these registers are modified but are NOT operands." The compiler avoids assigning operands to clobbered registers and saves/restores them if live. `"memory"` = may read/write arbitrary memory. `"cc"` = condition codes modified.

**Investigation pattern — the uncovered register check:**
For each register that the asm body writes to via a hard-coded name (e.g., `"MOV r4, ..."`), verify that the register is covered by EITHER:
1. An operand constraint (the compiler assigned it and knows it's modified), OR
2. The clobber list (the compiler knows it's modified even though it's not an operand)

If a hard-coded register is in NEITHER, the compiler may assign an operand to that register or keep a live value there. When the asm body writes to it, the compiler's value is silently destroyed. This is the root cause of most optimization-dependent inline asm bugs.

**Write-before-read hazard in asm preambles:**
Many wolfSSL asm functions begin with a "preamble" that saves operand values to scratch registers before the main computation. Example:
```
"MOV  lr, %[in]\n\t"           // save %[in] to lr
"MOV  r0, %[table_ptr]\n\t"    // save table to r0 — but what if %[len] is in r0?
"MOV  r12, %[len]\n\t"         // reads %[len] — CORRUPTED if compiler put it in r0
```
The compiler is free to assign `%[len]` to r0 if r0 isn't clobbered and isn't bound to another operand. The preamble then overwrites r0 with the table pointer BEFORE reading `%[len]`, destroying the value.

**Investigation pattern:** Trace the first N instructions of any asm block. For each hard-coded register write, ask: "Could the compiler have assigned an unread operand to this register?" If yes, there is a potential write-before-read hazard. This hazard manifests at higher optimization levels when the compiler's register allocator reorders assignments.

## WOLFSSL_NO_VAR_ASSIGN_REG — Compiler Binding Architecture

This mechanism is the most common source of inline assembly bugs in wolfSSL's cross-compiler support.

**What register binding does:** GCC's `register int x __asm__("r4")` forces variable `x` into r4. wolfSSL's generated code uses this as the DEFAULT path — parameter 0 → r0, parameter 1 → r1, etc.

**Why IAR/Keil can't use it:** This syntax is GCC-specific. When `WOLFSSL_NO_VAR_ASSIGN_REG` is defined (automatically for IAR, Keil), the generated code drops register binding and uses generic operand constraints instead. The compiler freely assigns ANY available register to each operand.

**The fundamental contract change:**
- **With binding:** Register N = Parameter N is GUARANTEED. The generator knows this and can safely hard-code register names in the asm body.
- **Without binding:** The compiler assigns registers freely. Hard-coded register names in the asm body no longer correspond to specific parameters. If the asm body writes to r4, it might be destroying operand %[in] — which the compiler may have assigned to r4.

**How the generator builds clobber lists — and where it breaks:**
The Ruby generator's `asm_vars()` method skips register N from the clobber list if parameter N exists (register N IS the parameter under binding). This is correct for the binding path but wrong for the no-binding path, where parameter N can be in any register.

**Investigation pattern for NO_VAR_ASSIGN_REG bugs:**
1. Read the `_c.c` file's `#ifdef WOLFSSL_NO_VAR_ASSIGN_REG` section — check whether the operand constraints differ between the two paths (e.g., constants may be `"+r"` in one path and `"r"` in the other)
2. Read the generator's `asm_vars()` method in the scripts repo's architecture module (e.g., `asm/thumb2/thumb2_c.rb`) — look for the skip logic that filters registers from the clobber list
3. In the generated clobber list, verify that ALL hard-coded registers used in the asm body are covered by either an operand or a clobber — not just registers that "correspond to" parameters by index
4. Compare the clobber lists of multiple functions in the same `_c.c` file — functions with more parameters will have more registers skipped from the clobber list, making them more vulnerable
5. Check for hardcoded stack loads (`LDR rX, [sp, #offset]`) in the NO_VAR_ASSIGN_REG path — these assume a specific stack layout from the standalone `.S` function prolog (callee-saved register count × 4 bytes). In inline `_c.c` mode, the compiler manages the stack frame and may push different registers, making the offsets incorrect. Look for the generator's `get_param_stack()` or `param_stack()` method — if it computes offsets from `@stack_regs_size`, those offsets are only valid for the standalone path. The proper fix is to use operand references (`%[name]`) instead of stack loads.

## Register Pressure Analysis for Inline Assembly

When a clobber list bug is found, the obvious fix is "add the missing registers to the clobber list." But on register-constrained architectures, this may not be feasible.

**ARM register budget:** r0-r12 = 13 GPRs (r13=SP, r14=LR, r15=PC are reserved; r12 and lr often used as clobbers). For any inline asm block: `operand_count + clobber_count <= available_GPRs`. If this doesn't balance, the compiler cannot allocate registers — it will error or silently produce incorrect code.

**Investigation pattern — verify fix feasibility:**
When proposing "add rX to the clobber list" as a fix:
1. Count the current operands (output `"+r"` / `"=r"` + input `"r"`)
2. Count the current clobber registers
3. Add the proposed new clobbers
4. Verify: `operand_count + new_clobber_count <= 12` (approximate ARM limit)
5. If the budget doesn't balance, the fix requires restructuring the asm block (splitting into multiple blocks, passing values via memory instead of registers, or reducing the operand count)

**Thumb-2 additional constraint:** Some 16-bit Thumb instruction encodings can only access r0-r7 ("low registers"). If the asm body uses these narrow instructions, the compiler's choices for operand registers are further constrained to r0-r7, increasing pressure.

**Register saturation:** When `operand_count + clobber_count` equals the total available GPRs, ALL registers are accounted for — the clobber list is technically correct because every register is either an operand or clobbered. In this case, the primary risk is NOT missing clobbers but **preamble ordering hazards**: the asm preamble reads some operands and writes others using hard-coded names, and the compiler has no visibility into that ordering since it's inside the asm block. The compiler correctly knows every register may be modified, but it doesn't know WHEN each modification happens relative to the operand reads. Identify register saturation early — it shifts the investigation from "which registers are missing from the clobber list" to "does the preamble read all operands before any hard-coded register writes."

## Alignment Investigation Pattern

Alignment issues are common in cross-compiler assembly work. The key distinction is WHERE the aligned variable lives.

**How to verify alignment behavior:**
1. Grep for `XALIGNED` in `wolfssl/wolfcrypt/types.h` to see how each compiler implements the alignment macro
2. Determine whether the aligned variable is **stack-allocated** (automatic/local variable) or **static/global**
3. **Static/global arrays** with `XALIGNED(16)`: alignment is controlled by the linker section placement — generally reliable across all compilers
4. **Stack (automatic) variables** with alignment attributes: behavior is compiler-specific. Different compilers have different maximum alignment guarantees for stack variables. Do NOT claim specific limits (e.g., "8-byte max") without citing the compiler's documentation.

**Investigation pattern:** When alignment is involved:
- (a) Grep `XALIGNED` in `types.h` to verify the macro expansion for the customer's compiler
- (b) Find the variable in the source and determine: is it stack-allocated or static/global?
- (c) If stack-allocated and the issue is alignment-related, note that compiler-specific limits may apply. Verify via `web_fetch` to the compiler's documentation, or tag the claim as `[UNVERIFIED — requires [compiler] documentation]`
- (d) Check if the generated code's Ruby generator uses `constanta()` for the variable — this method generates `static const` declarations with `XALIGNED`, which are linker-controlled
- (e) Check `wolfssl/wolfcrypt/types.h` for any compiler-specific overrides or caps on alignment macros (e.g., `WOLFSSL_USE_ALIGN`, `WOLFSSL_ALIGN()`)

## ARM ABI Investigation Pattern

ARM calling conventions and ABI details are relevant when debugging register allocation, stack layout, or cross-function assembly behavior.

**Where to look:**
- ARM AAPCS (Procedure Call Standard for the Arm Architecture) defines register usage, stack alignment, and calling conventions
- `developer.arm.com` is in the allowed web_fetch domains — fetch the AAPCS when you need to verify ABI claims

**Investigation pattern:** Do NOT state ABI facts from memory — fetch the AAPCS from `developer.arm.com`. Key areas: caller-saved vs callee-saved registers, stack alignment at function entry, Thumb-2 interworking. The scripts repo's architecture modules define register allocation (`@stack_regs`, `@param_regs`) — compare against the AAPCS.

## Compiler Optimization Interaction Patterns

Inline assembly interacts with compiler optimizers differently across toolchains. This is a common source of "works at -O0 but breaks at -O2" bugs.

**Why -O0 vs -O2 is the signature symptom of clobber/constraint bugs:**
At -O0, compilers typically assign operands to registers in declaration order and avoid aggressive register reuse. This naive allocation often HAPPENS to match the generator's assumptions (register N = parameter N). At -O2, the compiler optimizes register allocation — it may assign operands in any order, reuse registers across operations, and allocate based on liveness analysis. This exposes register conflicts that were latent at -O0: an operand assigned to a register that the asm body also writes via hard-coded instructions will be silently corrupted.

**Key areas of divergence:**
- **Constant propagation through `const` data** — Compilers may assume `const` data never changes and optimize accordingly. Inline assembly that reads `const` data may be affected if the compiler reorders or eliminates the data access.
- **Register allocation around `asm volatile` blocks** — The `volatile` qualifier prevents the compiler from eliminating or reordering the assembly block, but data loaded BEFORE the block (e.g., via register variables) may still be subject to optimization.
- **Clobber lists** — The `"memory"` clobber tells the compiler that the assembly may read/write arbitrary memory, preventing it from caching values across the boundary. But data loaded into registers before the asm block may not be covered.

**Why `volatile` on data masks but does not fix clobber bugs:**
Adding `volatile` to data declarations changes register allocation pressure (the compiler can no longer propagate values or reorder accesses), which may cause operands to land in different registers — accidentally avoiding the conflicting register. This is fragile: any change to surrounding code or compiler version can re-trigger the conflict. The real fix is correcting the clobber list or operand constraints.

**Investigation pattern:** When a ticket reports optimization-dependent behavior:
1. Read the `asm volatile` block's **clobber list** and **operand list** in the generated code
2. Check whether data accessed by the assembly is loaded BEFORE the asm block (via register variables or pointer assignments) or passed AS input/output operands
3. Check what the compiler's optimizer is allowed to assume about the data:
   - Is it declared `const`? (compiler may assume it doesn't change)
   - Is it declared `volatile`? (compiler must re-read on every access)
   - Is it passed as an operand with the right constraint? (compiler knows the dependency)
4. Read the generator's `asm_vars()` or equivalent method to understand how the operand list and clobber list are constructed
5. Perform the **uncovered register check** (see "GCC Extended Inline Assembly Correctness Model" above) — this is almost always the root cause of optimization-dependent asm bugs
6. If the customer proposes `volatile` as a workaround, acknowledge that it may work empirically but explain that it masks the root cause — a clobber list or operand constraint correction is the proper fix
7. Do NOT recommend compiler-specific optimization pragmas or workarounds without verifying they exist in the compiler's documentation. If you cannot verify, present the suggestion with an explicit `[UNVERIFIED — requires [compiler] documentation]` tag
